# 控制流程页面无限滚动 — 前端实现关键点

## 一、实现步骤

### 1. API 层支持分页参数

`src/api/request.js` 的 `get()` 增加 `params` 参数，拼接 query string：

```js
export const get = (path, params) => {
  const query = params ? '?' + new URLSearchParams(params).toString() : ''
  return request(`${path}${query}`)
}
```

`src/api/matrix.js` 透传分页参数：

```js
export const fetchRiskList = (code, params) =>
  get(`/matrix/process/${code}/risks`, params)
```

### 2. Composable 改造为无限滚动逻辑（useRiskList.js）

核心状态：

| 状态 | 说明 |
|------|------|
| `riskItems` | 累积所有已加载的数据（跨页追加） |
| `columns` | 列定义，仅首页赋值一次 |
| `pageNum` | 当前请求页码，加载成功后自增 |
| `loading` | 是否正在加载，由 `van-list` 通过 `v-model` 控制 |
| `finished` | 是否全部加载完毕 |

`loadMore` 关键逻辑：

```js
async function loadMore() {
  if (finished.value) return        // 已结束则跳过
  loading.value = true
  try {
    const data = await fetchRiskList(code, { pageNum: pageNum.value, pageSize: PAGE_SIZE })
    if (pageNum.value === 1) {      // 仅首页设置列定义和标题
      processName.value = data.processName
      columns.value = data.columns ?? []
    }
    riskItems.value.push(...data.riskItems)    // 追加而非替换
    if (riskItems.value.length >= data.total) {
      finished.value = true         // 已加载数 >= 总数，停止
    } else {
      pageNum.value++
    }
  } catch (e) {
    error.value = e.message
  } finally {
    loading.value = false           // 无论成功失败都必须释放
  }
}
```

### 3. 视图层使用 van-list

```html
<van-list
  v-model:loading="loading"
  :finished="finished"
  finished-text="已全部加载"
  @load="loadMore"
>
  <risk-row v-for="(item, i) in riskItems" :key="i" ... />
</van-list>
```

`van-list` 的触发机制：

- 组件挂载后，若内容未撑满滚动容器，**自动触发一次** `@load`（首次加载）
- 用户滚动到距底部 `offset`（默认 300px）时，继续触发 `@load`
- `loading=true` 期间不会重复触发
- `finished=true` 后停止触发，显示 `finished-text`

**为什么初始会连续请求多页？**

每次 `loading` 变为 `false` 后，`van-list` 都会检查：

> 滚动容器底部 距离 可视区底部 是否 ≤ `offset`（默认 300px）

若满足条件就再触发一次 `@load`。初始时列表为空，内容高度为 0，远小于可视区高度，因此会连续自动加载，直到内容撑满可视区 + 300px 缓冲区为止。

以本项目为例（每行约 40px，移动端可视区约 560px）：

```
第1页（5条）：内容 ≈ 200px < 560px → 距底部 0px ≤ 300px → 自动触发第2页
第2页（10条）：内容 ≈ 400px < 560px → 距底部 0px ≤ 300px → 自动触发第3页
第3页（15条）：内容 ≈ 600px > 560px → 距底部 ≈ 340px > 300px → 停止自动触发
```

实际连续加载几页，由以下因素共同决定：

| 因素 | 影响 |
|------|------|
| 每行高度 | 行越高，越快撑满，自动加载页数越少 |
| 可视区高度 | 屏幕越高，需要更多页才能撑满 |
| `offset` 值 | 默认 300px，值越大，停止越晚 |
| `pageSize` | 每页条数越多，越快撑满 |

这是 `van-list` 的设计意图：**自动预加载到内容充满屏幕**，之后才由用户滚动驱动后续加载。

### 4. pageSize 与 pageNum 的变化时机

**pageSize** — 永远不变。定义为模块级常量 `PAGE_SIZE = 5`，每次请求固定传递，仅作配置项使用。

**pageNum** — 有两个时机发生变化：

| 时机 | 变化 |
|------|------|
| `useRiskList` 初始化时 | 初始值 `pageNum = 1` |
| 每次 `loadMore` 请求成功且数据未全部加载完时 | `pageNum++` |

完整流程示例（总数据 12 条，每页 5 条）：

```
初始:  pageNum=1

@load → 请求 pageNum=1 → 返回5条，total=12 → 5<12  → pageNum++ → pageNum=2
@load → 请求 pageNum=2 → 返回5条，total=12 → 10<12 → pageNum++ → pageNum=3
@load → 请求 pageNum=3 → 返回2条，total=12 → 12>=12 → finished=true（pageNum 不再变化）
```

**注意**：`pageNum` 只在请求**成功后**才自增。请求失败时 `pageNum` 保持不变，`van-list` 下次重试仍使用同一页码重新请求，保证数据不跳页。

## 二、Bug 记录

### Bug 1：页面一直转圈，无数据，请求未发出

**现象**：进入控制流程页面后，一直显示加载动画，列表为空，接口未被调用。

**原因**：`van-list` 被包裹在 `v-else-if="columns.length"` 条件内。初始状态 `columns` 为空，`van-list` 根本不渲染，因此永远不会触发 `@load`，请求无法发出，`columns` 也永远不会被赋值——死锁。

```html
<!-- 错误写法：van-list 被条件阻断，永远不渲染 -->
<div v-else-if="columns.length" class="risk-card">
  <van-list @load="loadMore">...</van-list>
</div>
<van-loading v-else />  <!-- 一直走这里 -->
```

**修复**：`van-list` 始终渲染，仅表头用 `v-if` 控制：

```html
<!-- 正确写法 -->
<div v-else class="risk-card">
  <div v-if="columns.length" class="table-header">...</div>
  <van-list @load="loadMore">...</van-list>
</div>
```

---

### Bug 2：数据加载成功，但底部"加载中"持续显示

**现象**：第一页数据正常展示，底部加载动画一直不消失。

**原因**：修复 Bug 1 时额外加了 `onMounted(loadMore)`，与 `van-list @load` 形成两个驱动源，相互干扰：

1. `van-list` 挂载 → 触发 `@load` → `loadMore` 运行 → `loading = true` → 第一页请求发出
2. `onMounted` 也触发 `loadMore` → `loading` 已为 `true` → 被守卫 `if (loading.value) return` 拦截
3. 第一页完成 → `loading = false`
4. `van-list` 再次触发 `@load` → `loadMore` 运行 → `loading = true` → 第二页请求发出
5. 两个驱动源导致 `loading` 状态与 `van-list` 内部预期不一致，`finished` 无法在正确时机被设置，加载动画永不消失

**修复**：
- 移除 `onMounted(loadMore)`，由 `van-list` 统一管理所有加载时机（包括首次加载）
- 移除 `loadMore` 内的 `if (loading.value) return` 守卫——`van-list` 已内置防重入，冗余守卫会拦截合法调用

```js
// 修复后：只保留 finished 检查，不手动守卫 loading
async function loadMore() {
  if (finished.value) return
  loading.value = true
  // ...
}
```

## 三、设计原则总结

| 原则 | 说明 |
|------|------|
| `van-list` 必须始终渲染 | 不能被 `v-if` / `v-else-if` 阻断，否则 `@load` 无法触发 |
| 由 `van-list` 统一管理加载时机 | 不要在 `onMounted` 中重复触发 `loadMore`，避免多驱动源冲突 |
| 不在 `loadMore` 中手动守卫 `loading` | `van-list` 已保证 `loading=true` 期间不重复触发 `@load` |
| `finally` 必须释放 `loading` | 无论成功或异常，`loading = false` 让 `van-list` 知道本次加载已结束 |
| `riskItems` 追加而非替换 | 分页数据用 `push` 累积，用赋值会清空已加载内容 |
